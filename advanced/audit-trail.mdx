## Overview

DRAIT Mini-MES maintains a comprehensive audit trail that logs all critical operations across the system. Every create, update, delete, and state change is recorded with full before/after snapshots.

## Audit Data Model

The `AuditLog` model captures all auditable events:

```prisma
model AuditLog {
  id         String          @id @default(cuid())
  companyId  String
  userId     String?         // Can be null for system events
  entityType AuditEntityType
  entityId   String
  action     AuditActionType
  before     Json?           // State before change
  after      Json?           // State after change
  metadata   Json?           // Additional context
  createdAt  DateTime        @default(now())
  company    Company         @relation(...)
  user       User?           @relation(...)

  @@index([companyId, createdAt])
  @@index([companyId, entityType, entityId])
  @@index([companyId, action])
}
```

## Entity Types

The `AuditEntityType` enum defines what can be audited:


## What Gets Audited

### User Actions

| Operation | Entity Type | Action Type | Before/After |
|-----------|-------------|-------------|-------------|
| User created | `USER` | `CREATE` | `after` = new user data |
| User updated | `USER` | `UPDATE` | `before` + `after` = changed fields |
| User activated/deactivated | `USER` | `UPDATE` | `before` + `after` = `isActive` |
| Role changed | `USER` | `ROLE_CHANGE` | `before` + `after` = role |
| Password reset | `USER` | `PASSWORD_CHANGE` | `metadata` only |
| Login | `AUTH` | `LOGIN` | `metadata` = email |

### Work Order Actions

- Creation (`CREATE`)
- Status transitions (`STATUS_CHANGE`)
- Resource assignments (`ASSIGN`)
- Delivery and closure (`DELIVERY_CLOSE`)

### Material & Inventory

- Material creation and updates
- Stock adjustments
- Consumption tracking

### Client & Quotation

- Client data changes
- Quotation status changes
- Approval/rejection events

## Creating Audit Logs

### AuditService Interface

The `AuditService` provides a simple logging interface:

```typescript
interface CreateAuditEntryInput {
  companyId: string;           // Required: tenant isolation
  userId?: string | null;      // Optional: who performed the action
  entityType: AuditEntityType; // What type of entity
  entityId: string;            // Which specific entity
  action: AuditActionType;     // What happened
  before?: Prisma.InputJsonValue;  // State before (optional)
  after?: Prisma.InputJsonValue;   // State after (optional)
  metadata?: Prisma.InputJsonValue; // Extra context (optional)
}
```

### Basic Usage

From `apps/backend/src/modules/audit/audit.service.ts`:

```typescript
@Injectable()
export class AuditService {
  constructor(private readonly prisma: PrismaService) {}

  async log(input: CreateAuditEntryInput) {
    return this.prisma.auditLog.create({
      data: {
        companyId: input.companyId,
        userId: input.userId ?? null,
        entityType: input.entityType,
        entityId: input.entityId,
        action: input.action,
        before: input.before,
        after: input.after,
        metadata: input.metadata
      }
    });
  }
}
```

## Querying Audit Logs

### API Endpoint

From `apps/backend/src/modules/audit/audit-logs.controller.ts`:

```typescript
@Controller('audit-logs')
@UseGuards(JwtAuthGuard, RolesGuard)
export class AuditLogsController {
  @Get()
  @Roles('DUENO', 'SUPERVISOR', 'ADMIN')
  async findAll(
    @CurrentUser() user: JwtUser,
    @Query('entityType') entityType?: string,
    @Query('action') action?: string,
    @Query('limit') limit?: string
  ) {
    const take = Math.min(Math.max(Number(limit ?? 50), 1), 200);

    return this.prisma.auditLog.findMany({
      where: {
        companyId: user.companyId,
        entityType: entityType && entityType in AuditEntityType 
          ? (entityType as AuditEntityType) 
          : undefined,
        action: action && action in AuditActionType 
          ? (action as AuditActionType) 
          : undefined
      },
      include: {
        user: {
          select: {
            id: true,
            fullName: true,
            role: true,
            email: true
          }
        }
      },
      orderBy: { createdAt: 'desc' },
      take
    });
  }
}
```

### Query Examples

**Get all audit logs (last 50):**

**Filter by entity type:**

## Access Control

Only privileged roles can view audit logs:

- **DUENO** - Full access
- **SUPERVISOR** - Full access
- **ADMIN** - Full access
- **OPERARIO** - No access

<Info>
Audit logs are always scoped to the user's company (`companyId`) to ensure multi-tenant isolation.
</Info>

## Retention and Compliance

### Current Implementation

- Audit logs are stored indefinitely in PostgreSQL
- No automatic cleanup or archival
- All logs are tenant-isolated via `companyId`

### Recommended Retention Policies

For compliance and performance, consider implementing:

1. **Hot storage**: Last 90 days in primary database
2. **Warm storage**: 91-365 days in compressed format
3. **Cold storage**: 1+ years in archival system
4. **Legal holds**: Preserve specific records indefinitely

### Query Limits

API enforces reasonable limits:
- Default: 50 records
- Minimum: 1 record
- Maximum: 200 records per request

## Best Practices

1. **Always provide context**: Use `metadata` to include relevant details (e.g., reason for change)
2. **Capture before/after**: For updates, always include both `before` and `after` states
3. **Log actor**: Include `userId` when available to track who made changes
4. **Use specific actions**: Choose the most specific `AuditActionType` (prefer `ROLE_CHANGE` over generic `UPDATE`)
5. **Don't log secrets**: Never include passwords, tokens, or sensitive data
6. **Tenant isolation**: Always filter by `companyId` in queries

## Overview

DRAIT Mini-MES uses PostgreSQL 16 with Prisma ORM for database management. The system includes a comprehensive schema for manufacturing execution with support for multi-tenancy, work orders, resources, materials, and audit logging.

## Database Architecture

### Technology Stack

- **Database**: PostgreSQL 16 Alpine
- **ORM**: Prisma 5.22
- **Client**: @prisma/client
- **Migration Tool**: Prisma Migrate

### Schema Location

The Prisma schema is located at `apps/backend/prisma/schema.prisma:1`.

## Initial Database Setup

<Steps>
  <Step title="Ensure database container is running">
    Verify the PostgreSQL container is healthy:
    
    ```bash
    docker compose ps drait-db
    ```
    
    Expected output should show "healthy" status.
  </Step>
  
  <Step title="Generate Prisma client">
    Generate the Prisma client TypeScript types:
    
    ```bash
    docker compose exec drait-backend npm run prisma:generate
    ```
    
    This runs `prisma generate` and creates typed database client code.
  </Step>
  
  <Step title="Run enum normalization">
    DRAIT uses Spanish enum values. Normalize existing enums if migrating:
    
    ```bash
    docker compose exec drait-backend npm run prisma:normalize-enums
    ```
    
    This executes `apps/backend/prisma/enum-rename.sql` to rename English enum values to Spanish.
  </Step>
  
  <Step title="Apply database migrations">
    Push the Prisma schema to the database:
    
    ```bash
    docker compose exec drait-backend npm run prisma:migrate
    ```
    
    This runs:
    1. `prisma:normalize-enums` (enum fixes)
    2. `prisma db push` (schema sync)
  </Step>
  
  <Step title="Seed demo data (optional)">
    Populate the database with sample manufacturing data:
    
    ```bash
    docker compose exec drait-backend npm run prisma:seed
    ```
    
    See [Demo Data Seeding](#demo-data-seeding) below for details.
  </Step>
</Steps>

## Database Schema

### Core Entities

The schema includes 13 main models:

| Model | Description | Key Fields |
|-------|-------------|------------|
| **Company** | Multi-tenant organization | `id`, `name`, `taxId`, `settings` |
| **User** | System users with roles | `email`, `role`, `passwordHash` |
| **Client** | Customers | `name`, `contact`, `email` |
| **Quotation** | Sales quotes | `code`, `status`, `estimatedCost` |
| **WorkOrder** | Manufacturing orders | `code`, `status`, `priority`, `commitmentDate` |
| **Resource** | People and machines | `type`, `status`, `sector` |
| **Material** | Inventory items | `stock`, `unitCost`, `category` |
| **OperationLog** | Work order events | `eventType`, `eventAt`, `note` |
| **MaterialConsumption** | Material usage tracking | `quantity`, `unitCostSnapshot` |
| **AuditLog** | System audit trail | `entityType`, `action`, `before`, `after` |

### Enums

All enums use Spanish values (see `apps/backend/prisma/schema.prisma:12`):

**UserRole**:
- `DUENO` (Owner)
- `SUPERVISOR`
- `OPERARIO` (Operator)
- `ADMIN`

**WorkOrderStatus**:
- `PENDIENTE` (Pending)
- `PLANIFICADA` (Planned)
- `EN_PROCESO` (In Progress)
- `PAUSADA` (Paused)
- `FINALIZADA` (Finished)
- `ENTREGADA` (Delivered)
- `CANCELADA` (Canceled)
- `RETRABAJO` (Rework)

**ResourceType**:
- `HUMANO` (Human)
- `MAQUINA` (Machine)

**ResourceStatus**:
- `DISPONIBLE` (Available)
- `OCUPADO` (Busy)
- `MANTENIMIENTO` (Maintenance)
- `FUERA_DE_LINEA` (Offline)

<Note>
  The enum normalization script (`apps/backend/prisma/enum-rename.sql`) automatically converts English enum values from previous migrations to Spanish.
</Note>

## Prisma Scripts

All Prisma commands are defined in `apps/backend/package.json:13`:

### prisma:generate

```bash
docker compose exec drait-backend npm run prisma:generate
```

Generates Prisma Client with TypeScript types based on the schema.

**When to run**:
- After modifying `schema.prisma`
- After pulling schema changes
- During Docker image build (automatic)

### prisma:normalize-enums

```bash
docker compose exec drait-backend npm run prisma:normalize-enums
```

Executes SQL script to rename enum values from English to Spanish.

**SQL file**: `apps/backend/prisma/enum-rename.sql:1`

**Transformations**:
- `OWNER` → `DUENO`
- `OPERATOR` → `OPERARIO`
- `DRAFT` → `BORRADOR`
- `PENDING` → `PENDIENTE`
- `IN_PROGRESS` → `EN_PROCESO`
- And more...

### prisma:migrate

```bash
docker compose exec drait-backend npm run prisma:migrate
```

**Full command**: `npm run prisma:normalize-enums && prisma db push`

This is the main migration command that:
1. Normalizes enum values to Spanish
2. Syncs the Prisma schema to the database
3. Creates or updates tables, columns, indexes

<Note>
  This uses `prisma db push` instead of `prisma migrate dev` for faster prototyping. For production, consider using versioned migrations with `prisma migrate deploy`.
</Note>

### prisma:seed

```bash
docker compose exec drait-backend npm run prisma:seed
```

**Script**: `apps/backend/prisma/seed.ts:1`

Populates the database with comprehensive demo data.

### Inspecting the Database

<CodeGroup>
```sql List Tables
-- View all tables
\dt

-- Or via SQL
SELECT tablename FROM pg_tables 
WHERE schemaname = 'public' 
ORDER BY tablename;
```

```sql Table Schema
-- Describe a table
\d "WorkOrder"

-- Or via SQL
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_name = 'WorkOrder'
ORDER BY ordinal_position;
```

```sql Record Counts
-- Count records in all tables
SELECT 
  schemaname,
  tablename,
  n_live_tup AS row_count
FROM pg_stat_user_tables
ORDER BY n_live_tup DESC;
```

```sql Check Enums
-- List all enum types
SELECT t.typname, e.enumlabel
FROM pg_type t
JOIN pg_enum e ON t.oid = e.enumtypid
WHERE t.typname = 'WorkOrderStatus'
ORDER BY e.enumsortorder;
```
</CodeGroup>

### Resetting the Database

<Warning>
  These commands will **permanently delete all data**. Use with caution!
</Warning>

<Steps>
  <Step title="Stop the backend">
    ```bash
    docker compose stop drait-backend
    ```
  </Step>
  
  <Step title="Reset database schema">
    ```bash
    docker compose exec drait-db psql -U drait -d drait_mes -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public;"
    ```
  </Step>
  
  <Step title="Re-run migrations">
    ```bash
    docker compose exec drait-backend npm run prisma:migrate
    ```
  </Step>
  
  <Step title="Re-seed data (optional)">
    ```bash
    docker compose exec drait-backend npm run prisma:seed
    ```
  </Step>
  
  <Step title="Restart backend">
    ```bash
    docker compose start drait-backend
    ```
  </Step>
</Steps>

**Alternative: Full container reset**

```bash
# Stop and remove containers with volumes
docker compose down -v

# Restart everything
docker compose up -d

# Run migrations and seed
docker compose exec drait-backend npm run prisma:migrate
docker compose exec drait-backend npm run prisma:seed
```

## Database Backup

See [Backup & Restore](/deployment/backup-restore) for comprehensive backup strategies.

### Quick Backup

```bash
# Backup to file
docker compose exec drait-db pg_dump -U drait drait_mes > backup_$(date +%Y%m%d_%H%M%S).sql

# Backup with compression
docker compose exec drait-db pg_dump -U drait drait_mes | gzip > backup_$(date +%Y%m%d_%H%M%S).sql.gz
```

### Quick Restore

```bash
# Restore from file
docker compose exec -T drait-db psql -U drait -d drait_mes < backup_20260313_143000.sql

# Restore from compressed file
gunzip -c backup_20260313_143000.sql.gz | docker compose exec -T drait-db psql -U drait -d drait_mes
```

## Troubleshooting

### Connection Issues

<Accordion title="Error: connection refused">
  **Cause**: Database container not running or not healthy.
  
  **Solution**:
  ```bash
  # Check container status
  docker compose ps drait-db
  
  # View logs
  docker compose logs drait-db
  
  # Restart database
  docker compose restart drait-db
  ```
</Accordion>

<Accordion title="Error: password authentication failed">
  **Cause**: Mismatch between `DATABASE_URL` and database credentials.
  
  **Solution**:
  1. Verify `.env` file has matching credentials:
     - `POSTGRES_USER` matches username in `DATABASE_URL`
     - `POSTGRES_PASSWORD` matches password in `DATABASE_URL`
     - `POSTGRES_DB` matches database name in `DATABASE_URL`
  
  2. Recreate containers:
     ```bash
     docker compose down
     docker compose up -d
     ```
</Accordion>

<Accordion title="Error: database does not exist">
  **Cause**: Database not created or wrong database name.
  
  **Solution**:
  ```bash
  # Create database manually
  docker compose exec drait-db psql -U drait -c "CREATE DATABASE drait_mes;"
  
  # Or recreate container
  docker compose down
  docker compose up -d drait-db
  ```
</Accordion>

For more issues, see [Troubleshooting](/deployment/troubleshooting#database-errors).

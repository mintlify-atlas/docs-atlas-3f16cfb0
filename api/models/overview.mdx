## Introduction

DRAIT Mini-MES uses **Prisma** as its ORM with **PostgreSQL** as the database. All data models follow consistent patterns for IDs, timestamps, soft deletes, and multi-tenancy.

---

## Common Field Patterns

### Primary Keys

All entities use **CUID** (Collision-resistant Unique Identifier) for primary keys:

```typescript
id: string // Example: "clx123abc456def789"
```

**Characteristics:**
- Automatically generated by Prisma
- URL-safe and sortable
- 25 characters long
- Globally unique across all tables

### Timestamps

All entities include automatic timestamp tracking:

<ResponseField name="createdAt" type="DateTime">
  Automatically set when entity is created
  
  ```typescript
  createdAt: DateTime @default(now())
  ```
</ResponseField>

<ResponseField name="updatedAt" type="DateTime">
  Automatically updated on every modification
  
  ```typescript
  updatedAt: DateTime @updatedAt
  ```
</ResponseField>

**Format:** ISO 8601 timestamp
```json
"createdAt": "2026-03-13T10:00:00.000Z"
```

### Soft Deletes

Most entities use soft delete pattern with `isActive` flag:

<ResponseField name="isActive" type="Boolean">
  Indicates if entity is active (default: true)
  
  ```typescript
  isActive: Boolean @default(true)
  ```
</ResponseField>

**Benefits:**
- Preserves historical data for audit and reporting
- Maintains referential integrity
- Allows "undelete" functionality
- Supports compliance requirements

**Usage:**
- Deleted entities have `isActive: false`
- List queries typically filter by `isActive: true`
- Soft-deleted entities remain in database

## Multi-Tenancy

All primary entities include `companyId` for tenant isolation:

<ResponseField name="companyId" type="string">
  Foreign key to Company entity
  
  ```typescript
  companyId: string
  company: Company @relation(fields: [companyId], references: [id], onDelete: Cascade)
  ```
</ResponseField>

**Key Points:**
- Every user, client, quotation, work order, etc. belongs to a company
- JWT tokens include `companyId` claim
- API automatically filters by authenticated user's company
- Cross-tenant data access is prevented
- `onDelete: Cascade` ensures data cleanup

## Decimal Fields

Financial and quantity fields use Prisma `Decimal` type for precision:

```typescript
unitCost: Decimal @db.Decimal(12, 2)  // 12 total digits, 2 decimal places
quantity: Decimal @db.Decimal(12, 3)  // 3 decimal places for quantities
```

**Common Patterns:**
- **Currency**: `Decimal(12, 2)` - up to $9,999,999,999.99
- **Quantities**: `Decimal(12, 3)` - supports fractional quantities
- **Percentages**: Stored as decimals (0.85 = 85%)

**API Representation:**
```json
{
  "unitCost": 8.50,
  "quantity": 250.500,
  "estimatedCost": 15000.00
}
```

## Relationships

### Foreign Keys

Relationships use CUID foreign keys with referential actions:

```typescript
clientId: string
client: Client @relation(fields: [clientId], references: [id], onDelete: Restrict)
```

**Common Actions:**
- `onDelete: Cascade` - Delete children when parent is deleted
- `onDelete: Restrict` - Prevent deletion if children exist
- `onDelete: SetNull` - Set FK to null when parent is deleted

### Eager Loading

API endpoints may include related entities:

```json
{
  "id": "clxw123abc",
  "code": "OT-2026-001",
  "client": {
    "id": "clx789ghi",
    "name": "Acme Corporation"
  },
  "assignments": [
    {
      "resourceId": "clxr456def",
      "userId": "clxu789ghi"
    }
  ]
}
```

## Indexes

Performance-critical fields are indexed:

```typescript
@@index([companyId, name])      // Composite index
@@index([companyId, status])    // Status filtering
@@index([companyId, isActive])  // Soft delete queries
@@unique([companyId, code])     // Unique within company
```

**Common Index Patterns:**
- All queries include `companyId` (tenant isolation)
- Status fields for filtering work orders, quotations
- Name/search fields for autocomplete
- Timestamp fields for date range queries

---

## Unique Constraints

Certain fields must be unique within a company:

```typescript
@@unique([companyId, email])  // Email unique per company
@@unique([companyId, code])   // Code unique per company
```

**Examples:**
- User emails (unique per company, not globally)
- Work order codes (unique per company)
- Quotation codes (unique per company)
- Client names (not enforced, but recommended)

## Validation

API uses **class-validator** for request validation:

```typescript
export class CreateClientDto {
  @IsString()
  @MaxLength(120)
  name!: string;

  @IsOptional()
  @IsEmail()
  email?: string;
}
```

**Common Validators:**
- `@IsString()` - String type
- `@IsNumber()` - Numeric type
- `@IsEmail()` - Valid email format
- `@IsEnum(EnumType)` - Must be enum value
- `@MinLength(n)` - Minimum string length
- `@MaxLength(n)` - Maximum string length
- `@Min(n)` - Minimum numeric value
- `@IsOptional()` - Field is optional

## Entity Lifecycle

### Creation
1. Validate request DTO
2. Set default values (`isActive: true`, etc.)
3. Generate CUID for `id`
4. Set `createdAt` and `updatedAt` timestamps
5. Apply `companyId` from authenticated user
6. Create audit log entry

### Update
1. Validate request DTO
2. Update only provided fields
3. Auto-update `updatedAt` timestamp
4. Create audit log entry with before/after snapshot

### Soft Delete
1. Set `isActive: false`
2. Update `updatedAt` timestamp
3. Create audit log entry
4. Entity remains in database
5. Exclude from normal list queries

## Best Practices

### When Creating Entities

1. **Always include required fields** - Check DTO for required fields
2. **Validate data client-side** - Reduce API errors
3. **Use appropriate data types** - Numbers for quantities, strings for text
4. **Respect length limits** - Truncate or validate before submission

### When Querying

1. **Use filters** - Reduce result set size with status, search filters
2. **Implement pagination** - For large datasets
3. **Cache when appropriate** - Dashboard data can be cached short-term
4. **Select needed fields** - Avoid loading unnecessary relationships

### When Updating

1. **Send only changed fields** - PATCH endpoints accept partial updates
2. **Handle optimistic locking** - Check `updatedAt` for conflicts
3. **Validate before sending** - Ensure data meets constraints

### Data Integrity

1. **Use soft deletes** - Preserve historical data
2. **Respect relationships** - Don't delete entities with active children
3. **Maintain audit trail** - Audit logs track all changes
4. **Test cascades** - Understand cascade delete behavior
